# Write Changelogs

Guidelines, best practices, and a structured framework for generating maintainable and informative changelogs following Keep a Changelog standards with proper versioning.

## Changelog Standards

### 1. Format (Keep a Changelog)
- Use headings for versions
- Date of release
- Categories: Added, Changed, Deprecated, Removed, Fixed, Security
- Link to issues/pull requests

### 2. Content Guidelines
- Focus on user-facing changes
- Be specific about changes
- Include migration instructions for breaking changes
- Credit contributors
- Link to relevant documentation

### 3. Version Categories
- **Added**: New features
- **Changed**: Existing feature modifications
- **Deprecated**: Soon-to-be removed features
- **Removed**: Removed features
- **Fixed**: Bug fixes
- **Security**: Vulnerability patches

### 4. Semantic Versioning
- MAJOR: Breaking changes
- MINOR: New features (backward compatible)
- PATCH: Bug fixes (backward compatible)

## When to Apply

Use this framework when:
- Documenting software releases
- Maintaining project changelogs
- Communicating changes to users
- Planning version releases

## Expected Output

When applying this framework, include:
1. Version header with date
2. Categorized changes
3. Links to related issues
4. Migration guide if needed
5. Contributor credits
6. Download/installation update info
